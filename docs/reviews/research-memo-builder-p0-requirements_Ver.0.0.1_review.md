# 2. P0必須修正

## P0-001：入力項目の「必須」と「初期値」の矛盾を解消する

### 問題

以下の項目は、表では **必須 Yes** になっています。

* `search.countPerQuery`
* `search.country`
* `search.searchLang`
* `search.uiLang`
* `search.extraSnippets`

一方で、要件文には以下のような表現があります。

* 未指定時は10としてもよい
* P0初期値は `JP`
* P0初期値は `ja`
* P0初期値は `ja-JP`
* P0初期値は `true`

これは、実装時に迷います。

### 修正案

以下のどちらかに統一してください。

#### 推奨案：`search` 配下は任意項目＋デフォルト補完

P0では、入力YAMLを軽く保つため、こちらを推奨します。

| 項目                     | 必須 | デフォルト   |
| ---------------------- | -: | ------- |
| `search.countPerQuery` | No | `10`    |
| `search.country`       | No | `JP`    |
| `search.searchLang`    | No | `ja`    |
| `search.uiLang`        | No | `ja-JP` |
| `search.extraSnippets` | No | `true`  |

この場合、要件として以下を追加するとよいです。

```text
P0-IN-XXX:
search 配下の項目は未指定時にP0既定値を補完する。
ただし、指定された場合は型と値範囲を検証する。
```

## P0-002：P0対象外の出力フラグは「省略可」にする

### 問題

現状では以下が `false` 固定になっています。

* `output.json`
* `output.runReport`
* `output.chatgptPrompt`

ただし、P0ではこれらを実装しないため、入力YAMLに毎回書かせる必要は薄いです。

### 修正案

以下の扱いがよいです。

| 項目                     | P0での扱い              |
| ---------------------- | ------------------- |
| `output.json`          | 省略可。`true` の場合は入力不正 |
| `output.runReport`     | 省略可。`true` の場合は入力不正 |
| `output.chatgptPrompt` | 省略可。`true` の場合は入力不正 |

修正文例：

```text
P0-IN-023:
output.json はP0では省略可とする。
指定された場合、falseのみ許可する。
trueの場合は入力不正として停止する。
```

## P0-003：全クエリ失敗時の終了判定を定義する

### 問題

現在は、

* 一部クエリ失敗時は継続
* すべてのクエリが失敗した場合は原因が分かるエラーメッセージを表示

までは定義されています。

ただし、以下が未確定です。

* `search-results.csv` を出すのか
* `research-memo.md` を出すのか
* exit code をどうするのか
* 「0件」と「API失敗」をどう区別するのか

### 修正案

以下のように決めるのがよいです。

| ケース                    | 処理                    | exit code |
| ---------------------- | --------------------- | --------: |
| 入力不正 / APIキー未設定 / 出力失敗 | 即停止                   |       `1` |
| 一部クエリ失敗、1件以上のクエリ成功     | CSV/Markdownを出力し、警告表示 |       `0` |
| 全クエリがAPI失敗             | 正常出力扱いにしない。エラー表示して停止  |       `1` |
| 全クエリ成功、検索結果0件          | 空CSVヘッダー＋Markdownを出力  |       `0` |

追加要件案：

```text
P0-ERR-020:
すべてのクエリがAPI失敗した場合は、正常なリサーチ結果として扱わず、exit code 1で終了する。

P0-ERR-021:
API失敗と検索結果0件は区別する。
検索結果0件は正常な検索結果として扱い、CSVヘッダーとMarkdownを出力する。
```

## P0-004：Brave Search APIのパラメータ変換を明記する

### 問題

入力YAMLでは camelCase です。

```yaml
countPerQuery
searchLang
uiLang
extraSnippets
```

一方で、API呼び出し時にはAPI側のパラメータ名へ変換する必要があります。

### 修正案

検索要件に以下を追加してください。

```text
P0-SEA-023:
内部入力項目とBrave Search APIパラメータの対応は以下とする。

- search.countPerQuery -> count
- search.country -> country
- search.searchLang -> search_lang
- search.uiLang -> ui_lang
- search.extraSnippets -> extra_snippets
```

これを入れると、`BraveSearchClient` 実装時の迷いが減ります。

## P0-005：検索結果URLへ直接アクセスしないことを明文化する

### 問題

安全制約として、以下はすでに明確です。

* note非公式APIを使わない
* note本文スクレイピングをしない
* 有料部分取得をしない

ただし、実装者視点では「検索結果URLをfetchして本文を見に行く」余地がまだ残ります。

### 修正案

安全制約に以下を追加してください。

```text
P0-SAFE-001:
P0ではBrave Search APIレスポンスのみを利用する。
検索結果URLに対してHTTP GET等の追加アクセスは行わない。

P0-SAFE-002:
P0では記事本文、HTML、OGP、価格表示、本文中の無料/有料境界を取得しない。
```

これはかなり重要です。
P0の安全ラインが一段はっきりします。

## P0-006：Markdownの「似たタイトル」欄にスニペットを含める

### 問題

P0の目的は、15分程度で既存記事リサーチを確認できる下準備を作ることです。

しかし、`research-memo.md` の「2. 似たタイトルがあるか」では、現状の要件だと以下が中心です。

* タイトル
* URL
* 媒体
* キーワード

CSVにはスニペットがありますが、Markdown側にも最低限のスニペットがないと、人間レビューが少し弱くなります。

### 修正案

`P0-MD-004` を以下のように修正するとよいです。

```text
P0-MD-004:
検索結果のタイトル、URL、媒体、キーワード、スニペットを一覧化する。
表示順は、重複排除後の取得順とする。
```

また、出力順も明記してください。

```text
P0-MD-012:
Markdown上の検索結果表示順は、platforms順、keywords順、rank順を基本とする。
ただし、URL重複排除により除外された結果は表示しない。
```

# 3. P1推奨修正

## P1-001：CSVのBOM有無を明記する

CSVをExcelで開く想定があるなら、`UTF-8 with BOM` の方が文字化けしにくいです。

現在は単に「UTF-8」となっています。

修正候補：

```text
P0-CSV-010:
CSVはUTF-8 with BOMで出力する。
MarkdownはUTF-8 without BOMで出力する。
```

ただし、CLIやテストの扱いやすさを優先するなら、BOMなしでも問題ありません。
どちらにするかだけ決めておくのが重要です。

---

## P1-002：既存ファイルの上書き方針を定義する

現状では、同じ `output.dir` に既存ファイルがある場合の扱いが未定義です。

修正候補：

```text
P0-OUT-007:
P0では同名の `search-results.csv` と `research-memo.md` が存在する場合、確認なしで上書きする。
```

または、

```text
P0-OUT-007:
P0では同名ファイルが存在する場合、処理を停止する。
```

おすすめは **上書き** です。
リサーチCLIは同じ入力で何度も再実行する可能性が高いためです。

---

## P1-003：`platforms` の重複扱いを定義する

現状では、以下のケースが未定義です。

```yaml
platforms:
  - name: note
    site: note.com
  - name: note2
    site: note.com
```

修正候補：

```text
P0-IN-032:
platforms[].site はtrim後・小文字化後に重複を禁止する。
重複がある場合は入力不正として停止する。
```

これはP0に入れてもよいですが、必須ではありません。
ただし、リクエスト数の制御を厳密にするならP0で入れる価値があります。

---

## P1-004：`.env` 読み込み位置を明記する

APIキー管理要件は十分ですが、どこから `.env` を読むかが未定義です。

修正候補：

```text
P0-KEY-012:
.env はプロジェクトルートから読み込む。
CLI実行時のカレントディレクトリはプロジェクトルートを前提とする。
```

## Cost

**評価：OK**

かなり良いです。

* 媒体上限：3件
* ページングなし

コスト制御としては十分です。

```text
P0では、入力バリデーション後の最大クエリ数を15件とする。
```

## Safety

**評価：P0修正必要**

安全方針はかなり良いです。

ただし、以下を明文化した方が強いです。

* Brave Search APIレスポンスだけを使う
* 検索結果URLへHTTPアクセスしない

CSVとMarkdownだけでP0目的は満たせます。

* Markdown上の検索結果並び順
* 0件時のMarkdown表示

* 部分失敗時の扱い
